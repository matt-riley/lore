# Releasing Lore

← [README](../README.md) · [Compatibility](compatibility.md) · [Support matrix](support-matrix.md) · [CONTRIBUTING](../CONTRIBUTING.md)

This document describes the versioning scheme, release-please workflow, and rollback/recovery guidance for Lore. It is the authoritative reference for cutting a release.

---

## Versioning

Lore follows [Semantic Versioning](https://semver.org/):

| Version component | When to bump |
|---|---|
| **MAJOR** (`1.x.x → 2.x.x`) | Breaking changes to a **supported** surface (hook signatures, supported tool names, DB schema removal). |
| **MINOR** (`0.1.x → 0.2.x`) | New supported surface, new experimental surface, backwards-compatible behaviour change. |
| **PATCH** (`0.1.0 → 0.1.1`) | Bug fixes, docs, internal refactors with no observable behaviour change. |

### `0.x` stability note

Until the `1.0.0` release, **experimental surfaces carry no SemVer promise**. A MINOR bump is sufficient to change or remove an experimental tool. See [support-matrix.md](support-matrix.md) for surface classifications.

The current version lives in `package.json`, but it is maintained by release-please. Tags and GitHub Releases are created by automation after a reviewed release PR merges.

## Pre-release checklist

Work through this list top-to-bottom before tagging. Every item must be green.

### 1. Tests and validation

```sh
npm test                   # all unit + smoke tests
npm run test:smoke         # smoke tests in isolation (confirm subprocess pipeline)
npm run validate-schema    # schema/config drift check
```

All must pass with zero failures. If any test is skipped (e.g., FTS5 unavailable locally) confirm the skip is expected and not a hidden failure.

### 2. Conventional commits are in place

- Merged PRs use Conventional Commit titles/messages such as `feat:`, `fix:`, `docs:`, `chore:`.
- Breaking changes include a `!` or `BREAKING CHANGE:` footer so release-please can cut the right version.
- The `Conventional PR title` workflow is green for the release PR and all merged changes leading into it.

### 3. Release PR looks correct

- `package.json` version bump is correct in the release PR.
- `CHANGELOG.md` entries generated by release-please are accurate and readable.
- Any noteworthy migration guidance is present before merge.

### 4. Compatibility doc current

- `docs/compatibility.md` reflects the correct Node minimum and any platform changes.
- The [Versioning section](compatibility.md#versioning) is consistent with the release you're cutting.

### 5. Support matrix current

- `docs/support-matrix.md` correctly reflects any surfaces that graduated (experimental → supported) or were deprecated/removed this cycle.

### 6. CI green on `main`

All CI jobs must be green on the commit you intend to tag. Check `.github/workflows/ci.yml` and `.github/workflows/release.yml` for the full job list.

## Release flow

Lore uses release-please. Maintainers do **not** hand-edit the changelog for routine releases, manually bump `package.json`, or create tags by hand.

### 1. Merge conventional commits into `main`

Every merged PR title should already follow Conventional Commits because `.github/workflows/conventional-pr-title.yml` enforces it.

### 2. Let release-please open or update the release PR

The release workflow runs on pushes to `main` and maintains a release PR with:

- the next version bump in `package.json`
- generated `CHANGELOG.md` entries
- the release commit title/body

If needed, you can also trigger the workflow manually from the Actions UI.

### 3. Review the release PR carefully

Before merging, confirm:

- the version bump matches the actual scope
- changelog sections are accurate
- any migration notes are present
- CI is green on the release PR

### 4. Merge the release PR

Merging the release PR is the release action. release-please will create the tag and publish the GitHub Release automatically.

## Post-release

1. **Verify the GitHub Release** is published and points to the correct tag.
2. **Smoke-test a clean install** from the tag:

   ```sh
   git clone <repo> lore-release-check
   cd lore-release-check
   git checkout v0.x.x
   npm test
   npm run validate-schema
   node scripts/dev-install.mjs --dry-run
   ```

3. **Check the next cycle** — release-please will reopen or refresh the next release PR automatically as new conventional commits land.

## Rollback and recovery

### Scenario 1 — release PR is wrong before merge

Do **not** hand-edit `package.json` or `CHANGELOG.md` on the release PR unless the release-please docs explicitly call for it.

Instead:

1. Close or update the release PR.
2. Merge corrective conventional commits to `main`.
3. Let release-please regenerate the release PR with the corrected notes/version.

### Scenario 2 — release published, users have already installed

Users install Lore by cloning the repository and running `scripts/dev-install.mjs`, which creates a symlink to the repo root. There is no registry publish step. Recovery options:

1. **Patch release** — fix the bug, cut a new patch release (`vX.Y.Z+1`). This is the preferred path for most issues.

2. **Pin to a previous tag** — instruct users to check out the last known-good tag:

   ```sh
   cd <lore-repo>
   git fetch --tags
   git checkout v0.x.y    # previous good tag
   ```

   Because `dev-install.mjs` symlinks the working tree, checking out the old tag is sufficient. No reinstall needed.

3. **Force-push a fixed main** — avoid unless the release was brand new and no one has yet pinned to the bad tag. Communicate clearly if you do this.

### Scenario 3 — DB schema migration causes data issues

This is the highest-risk scenario. The DB is at `~/.copilot/lore.db`.

1. **Run validation first** before taking any action:

   ```sh
   # In the Copilot CLI session
   memory_validate
   memory_doctor_report
   ```

2. **Restore from backup** if you have one. Lore does not automatically back up the DB — users should include `~/.copilot/lore.db` in their normal backup strategy.

3. **Re-derive from session-store** — if the DB is corrupted and there is no backup, the memory store can be partially rebuilt by re-running backfill tools against the raw `session-store.db` (which Lore never writes to). The rebuilt store will be missing any memories that were saved explicitly but not derivable from raw sessions.

4. **Fresh start** — delete `lore.db` and restart the Copilot CLI. Lore will create a clean DB and run all migrations from scratch. You lose all memories.

> **Migration safety rule**: all schema migrations in Lore are additive (add columns, add tables, add indexes). Destructive migrations are never applied silently. If a future release needs a breaking migration, it will be called out in the CHANGELOG as **BREAKING** and the pre-release checklist will include a migration test against a fixture DB.

### Scenario 4 — config schema changes break existing configs

1. **Check the error** — `memory_validate` will surface any config validation failures.
2. **Consult the CHANGELOG** — the entry for the release will list any config key changes.
3. **Update `lore.json`** per the migration notes in the CHANGELOG.
4. **Re-validate** — run `npm run validate-schema` (or `memory_validate` in-session) to confirm the updated config is valid.

## Required validation gates

The following must pass before any release is published. These are enforced by CI but must also be run locally as part of the checklist:

| Gate | Command | What it checks |
|---|---|---|
| Unit tests | `npm test` | All unit and smoke tests pass |
| Smoke tests | `npm run test:smoke` | End-to-end subprocess pipeline |
| Schema parity | `npm run validate-schema` | `lore.json` matches `lore.schema.json` |
| CI (push/release PR) | GitHub Actions | All of the above on the commit that will ship |

There are no optional gates. If any gate is red, the release does not ship.

---

## Release cadence

Lore has no fixed release cadence at the `0.x` stage. Releases happen when:

- A meaningful set of changes has accumulated and the checklist is green.
- A bug fix is urgent enough to warrant a patch release.

There is no time-boxed release train until the project matures to `1.x`.
